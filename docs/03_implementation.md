# 03. 구현 기록

## 1. 구현 개요

본 프로젝트는 스크린샷 이미지에서 텍스트를 추출하고, 추출된 텍스트를 저장한 뒤, 사용자가 검색어를 통해 다시 찾을 수 있도록 하는 웹 기반 아카이빙 서비스이다.

구현은 설계 문서에서 정의한 기능을 기준으로 단계적으로 진행하였다.

구현 순서는 다음과 같다.

1. Streamlit 기본 화면 구현
2. 이미지 업로드 및 미리보기 구현
3. OCR 텍스트 추출 기능 구현
4. SQLite 저장 및 조회 기능 구현
5. 키워드 검색 기능 구현

---

## 3. 파일별 구현 내용

### 3.1 app.py

`app.py`는 Streamlit 기반 화면 구성과 전체 사용자 흐름을 담당한다.

주요 역할은 다음과 같다.

- 서비스 제목 및 설명 표시
- 스크린샷 이미지 업로드
- 업로드한 이미지 미리보기
- OCR 실행 버튼 제공
- OCR 결과 확인 및 수정 영역 제공
- 저장 제목 입력
- 저장 버튼 처리
- 저장된 아카이브 목록 표시
- 검색어 입력 및 검색 결과 표시

`app.py`는 사용자가 직접 상호작용하는 화면을 담당하며, OCR, DB, 검색 로직은 별도 모듈에서 가져와 사용한다.

---

### 3.2 ocr_utils.py

`ocr_utils.py`는 이미지에서 텍스트를 추출하는 OCR 기능을 담당한다.

주요 함수는 다음과 같다.

| 함수명 | 설명 |
|---|---|
| load_reader() | EasyOCR Reader 객체를 로드한다. |
| extract_text_from_image(image) | 업로드된 이미지에서 텍스트를 추출한다. |

OCR 기능은 EasyOCR을 사용하여 구현하였다.  
한글과 영어가 포함된 스크린샷을 처리하기 위해 `["ko", "en"]` 언어 옵션을 사용하였다.

EasyOCR은 처음 실행 시 모델 로딩 시간이 발생할 수 있으므로, Streamlit의 cache 기능을 사용하여 Reader 객체를 반복적으로 로드하지 않도록 처리하였다.

---

### 3.3 db.py

`db.py`는 SQLite 데이터베이스 관련 기능을 담당한다.

주요 함수는 다음과 같다.

| 함수명 | 설명 |
|---|---|
| init_db() | archive 테이블을 생성한다. |
| save_archive(title, content) | 제목과 텍스트 내용을 DB에 저장한다. |
| get_all_archives() | 저장된 전체 아카이브 목록을 최신순으로 조회한다. |

저장되는 데이터는 다음과 같다.

| 컬럼명 | 설명 |
|---|---|
| id | 저장 항목의 고유 번호 |
| title | 저장 제목 |
| content | OCR로 추출 및 수정된 텍스트 |
| created_at | 저장 시각 |

이번 MVP에서는 이미지 원본을 저장하지 않고, OCR로 추출한 텍스트와 사용자가 수정한 텍스트만 저장한다.

---

### 3.4 search_utils.py

`search_utils.py`는 검색 기능을 담당한다.

현재 구현된 검색 방식은 키워드 검색이다.

주요 함수는 다음과 같다.

| 함수명 | 설명 |
|---|---|
| keyword_search(query, archives) | 검색어가 제목 또는 본문에 포함된 아카이브를 반환한다. |

키워드 검색은 사용자가 입력한 검색어가 저장된 아카이브의 제목 또는 본문에 포함되어 있는지 확인하는 방식이다.

예를 들어 저장된 텍스트에 “바이브코딩 과제 보고서 제출”이라는 문장이 있을 때, 사용자가 “바이브코딩”이라고 검색하면 해당 아카이브가 결과로 반환된다.

## 4. 기능별 구현 결과

### 4.1 이미지 업로드 및 미리보기

사용자는 `file_uploader`를 통해 png, jpg, jpeg 형식의 스크린샷 이미지를 업로드할 수 있다.

업로드된 이미지는 화면에 미리보기로 표시된다.

### 4.2 OCR 텍스트 추출

사용자가 OCR 실행 버튼을 누르면 업로드한 이미지에서 텍스트를 추출한다.

추출된 텍스트는 오른쪽 OCR 결과 확인 영역에 표시된다.

### 4.3 OCR 결과 수정

OCR 결과는 항상 정확하지 않을 수 있으므로, 사용자가 저장 전에 직접 수정할 수 있도록 `text_area`에 표시하였다.

실제 테스트에서 “젤”이 “절”로, “살몬”이 “살문”으로, “댓글”이 “대글”로 인식되는 사례가 있었다.  
따라서 OCR 결과를 바로 저장하지 않고 사용자 수정 단계를 둔 것이 필요하다는 점을 확인하였다.

### 4.4 SQLite 저장

사용자가 제목과 텍스트를 입력한 뒤 저장 버튼을 누르면 SQLite DB에 저장된다.

제목을 입력하지 않은 경우에는 본문 앞부분을 자동 제목으로 사용한다.  
저장할 본문이 비어 있는 경우에는 저장하지 않고 경고 메시지를 표시한다.

### 4.5 저장 목록 조회

저장된 아카이브는 최신순으로 화면에 표시된다.

각 항목은 제목, 저장 시각, 본문 내용을 포함한다.

### 4.6 키워드 검색

사용자는 검색어를 입력하여 저장된 아카이브를 검색할 수 있다.

검색어가 제목이나 본문에 포함되어 있으면 결과로 반환된다.  
검색어가 비어 있으면 경고 메시지를 표시하고, 결과가 없으면 “관련 결과가 없습니다.”라는 안내 메시지를 표시한다.

## 6. 현재 구현의 한계

현재 구현의 한계는 다음과 같다.

1. OCR 결과가 이미지 품질에 따라 부정확할 수 있다.
2. 현재 검색은 키워드 기반이므로 정확한 단어를 알아야 검색 가능하다.
3. 저장된 항목을 수정하거나 삭제하는 기능은 아직 없다.
4. 이미지 원본은 저장하지 않기 때문에 원본 스크린샷을 다시 확인할 수 없다.
5. 유사도 기반 검색은 아직 구현되지 않았다.

---

## 7. 다음 구현 계획

다음 단계에서는 키워드 검색의 한계를 보완하기 위해 유사도 검색 기능을 추가할 예정이다.

유사도 검색에서는 검색어와 저장된 텍스트를 벡터로 변환한 뒤, cosine similarity를 계산하여 관련성이 높은 결과를 반환한다.

이를 통해 사용자가 정확한 단어를 기억하지 못하고 “과제 조건”, “화장품 아이라이너”, “제출 관련 안내”처럼 뭉뚱그린 표현으로 검색해도 관련 내용을 찾을 수 있도록 개선할 계획이다.
