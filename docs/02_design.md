# 02. 시스템 설계

## 1. 시스템 개요

SnapText Archive는 사용자가 업로드한 스크린샷 이미지에서 텍스트를 추출하고, 추출된 텍스트를 저장한 뒤, 나중에 검색어와 유사한 내용을 다시 찾을 수 있도록 하는 웹 기반 아카이빙 서비스이다.

본 프로젝트는 과제 기간 안에 구현 가능한 MVP를 목표로 하며, 복잡한 모바일 앱 구조 대신 Streamlit 기반 단일 웹앱 구조로 설계한다.

초기 아이디어는 사용자가 스크린샷을 찍은 뒤 모바일 앱으로 바로 공유하면 앱이 자동으로 텍스트를 추출하고 저장하는 방식이었다. 그러나 모바일 공유 기능, 갤러리 접근 권한, 계정 관리, 클라우드 저장 기능까지 포함하면 구현 범위가 커지므로 이번 MVP에서는 웹 이미지 업로드 방식으로 범위를 축소하였다.

---

## 2. 설계 목표

본 프로젝트의 설계 목표는 다음과 같다.

1. 사용자가 쉽게 스크린샷 이미지를 업로드할 수 있도록 한다.
2. 이미지 속 텍스트를 OCR로 추출할 수 있도록 한다.
3. OCR 결과가 부정확할 수 있으므로 사용자가 직접 수정할 수 있도록 한다.
4. 확정된 텍스트를 SQLite DB에 저장한다.
5. 저장된 텍스트 목록을 조회할 수 있도록 한다.
6. 사용자가 정확한 문장을 기억하지 못하더라도 유사도 검색으로 관련 내용을 찾을 수 있도록 한다.
7. 기능별 코드를 분리하여 유지보수하기 쉬운 구조로 설계한다.

---

## 3. 전체 처리 흐름

```text
사용자
  ↓
스크린샷 이미지 업로드
  ↓
OCR 텍스트 추출
  ↓
OCR 결과 확인 및 수정
  ↓
SQLite DB 저장
  ↓
저장 목록 조회
  ↓
검색어 입력
  ↓
키워드 검색 또는 유사도 검색
  ↓
검색 결과 출력
````

전체 흐름은 이미지 업로드, 텍스트 추출, 저장, 검색의 네 단계로 구성된다.
사용자는 먼저 이미지를 업로드하고, 시스템은 OCR을 통해 이미지 속 텍스트를 추출한다. 이후 사용자는 추출된 텍스트를 확인하고 필요한 경우 수정한 뒤 저장한다. 저장된 텍스트는 이후 검색 대상이 되며, 사용자가 검색어를 입력하면 시스템은 저장된 텍스트 중 관련성이 높은 결과를 반환한다.

---

## 4. 화면 구성 설계

### 4.1 이미지 업로드 영역

이미지 업로드 영역은 사용자가 스크린샷 이미지를 업로드하는 영역이다.

주요 기능은 다음과 같다.

* png, jpg, jpeg 형식의 이미지 업로드
* 업로드한 이미지 미리보기
* OCR 실행 버튼 제공

이 영역은 사용자가 서비스를 처음 사용하는 진입점이므로 복잡한 설정 없이 이미지를 바로 업로드할 수 있도록 구성한다.

---

### 4.2 OCR 결과 확인 영역

OCR 결과 확인 영역은 이미지에서 추출된 텍스트를 보여주는 영역이다.

주요 기능은 다음과 같다.

* OCR로 추출된 텍스트 표시
* 사용자가 직접 텍스트 수정 가능
* 저장 제목 입력
* 저장 버튼 제공

OCR은 이미지 품질, 글자 크기, 배경색, 언어에 따라 결과가 부정확할 수 있다. 따라서 시스템이 추출한 텍스트를 사용자가 그대로 저장하는 것이 아니라, 저장 전에 직접 확인하고 수정할 수 있도록 설계하였다.

---

### 4.3 저장 목록 영역

저장 목록 영역은 지금까지 저장한 텍스트 아카이브를 확인하는 영역이다.

주요 기능은 다음과 같다.

* 저장된 항목 목록 조회
* 저장 제목 표시
* 저장 시간 표시
* 저장 내용 일부 표시

이 영역을 통해 사용자는 자신이 어떤 스크린샷 텍스트를 저장했는지 확인할 수 있다.

---

### 4.4 검색 영역

검색 영역은 사용자가 저장된 텍스트를 다시 찾기 위한 영역이다.

주요 기능은 다음과 같다.

* 검색어 입력
* 검색 버튼
* 검색 결과 목록 표시
* 유사도 점수 표시

사용자는 정확한 문장을 입력하지 않아도 된다. 예를 들어 저장된 텍스트가 “프로세스에 입각한 바이브코딩의 효과에 대해서 분석하고 정리하여 보고서를 제출하시오”일 때, 사용자가 “바이브코딩 과제 조건”처럼 뭉뚱그려 검색해도 관련 결과를 찾을 수 있도록 하는 것이 목표이다.

---

## 5. 데이터베이스 설계

이번 MVP에서는 SQLite를 사용한다. SQLite는 별도의 서버 설정이 필요 없고, 로컬 파일 기반으로 동작하기 때문에 작은 규모의 과제용 웹앱에 적합하다.

### 5.1 archive 테이블

| 컬럼명        | 타입      | 설명                |
| ---------- | ------- | ----------------- |
| id         | INTEGER | 기본키               |
| title      | TEXT    | 저장 항목 제목          |
| content    | TEXT    | OCR로 추출 및 수정된 텍스트 |
| created_at | TEXT    | 저장 시각             |

### 5.2 저장 데이터 예시

| id | title      | content                                        | created_at       |
| -- | ---------- | ---------------------------------------------- | ---------------- |
| 1  | 과제 안내 스크린샷 | 프로세스에 입각한 바이브코딩의 효과에 대해서 분석하고 정리하여 보고서를 제출하시오. | 2026-05-20 14:00 |
| 2  | 여행 준비물     | 여권, 충전기, 예약 확인서, 보조배터리 준비                      | 2026-05-20 15:30 |

### 5.3 이미지 원본 저장 여부

이번 MVP에서는 이미지 원본을 DB에 저장하지 않는다.

그 이유는 다음과 같다.

1. 이미지 원본을 저장하면 저장 용량이 커질 수 있다.
2. 스크린샷에는 개인정보가 포함될 수 있다.
3. 이번 과제의 핵심 기능은 이미지 보관이 아니라 텍스트 추출과 검색이다.
4. 구현 범위를 단순화하기 위해 텍스트 중심으로 저장한다.

따라서 이번 버전에서는 OCR로 추출한 텍스트와 사용자가 수정한 텍스트만 저장한다.

## 9. 검색 기능 설계

검색 기능은 두 단계로 구현한다.

### 9.1 1차 검색: 키워드 검색

1차 구현에서는 사용자가 입력한 검색어가 저장된 텍스트에 포함되어 있는지 확인하는 방식으로 검색한다.

예를 들어 저장된 텍스트에 “바이브코딩 과제 보고서 제출”이라는 문장이 있을 때, 사용자가 “바이브코딩”이라고 검색하면 해당 항목을 반환한다.

장점은 다음과 같다.

* 구현이 단순하다.
* 검색 속도가 빠르다.
* 초기 기능 검증에 적합하다.

한계는 다음과 같다.

* 사용자가 정확한 단어를 기억해야 한다.
* 같은 의미라도 표현이 다르면 검색이 어렵다.
* “보고서 제출 조건”처럼 뭉뚱그린 검색에는 약하다.

---

### 9.2 2차 검색: 유사도 검색

2차 구현에서는 검색어와 저장된 텍스트를 벡터로 변환한 뒤, cosine similarity를 계산하여 의미적으로 유사한 결과를 반환한다.

예를 들어 저장된 텍스트가 “프로세스에 입각한 바이브코딩의 효과에 대해서 분석하고 정리하여 보고서를 제출하시오”일 때, 사용자가 “바이브코딩 과제 조건”이라고 검색해도 관련성이 높은 항목을 찾을 수 있다.

장점은 다음과 같다.

* 정확한 단어를 기억하지 못해도 검색 가능하다.
* 의미적으로 비슷한 내용을 찾을 수 있다.
* 스크린샷 아카이빙 서비스의 핵심 가치와 잘 맞는다.

한계는 다음과 같다.

* 모델 로딩 시간이 필요할 수 있다.
* OCR 결과가 부정확하면 검색 품질이 떨어질 수 있다.
* 너무 짧은 텍스트에서는 유사도 계산 결과가 불안정할 수 있다.

## 10. OCR 기능 설계

OCR 기능은 업로드된 이미지에서 텍스트를 추출하는 역할을 한다.

이번 프로젝트에서는 구현 난이도와 실행 환경을 고려하여 Python 기반 OCR 라이브러리를 사용한다. OCR 결과는 이미지 품질에 따라 달라질 수 있으므로, 추출 결과를 사용자가 직접 수정할 수 있도록 설계한다.

OCR 처리 흐름은 다음과 같다.

```text
이미지 업로드
  ↓
이미지 객체 변환
  ↓
OCR 라이브러리로 텍스트 추출
  ↓
추출 텍스트 반환
  ↓
사용자가 확인 및 수정
```

OCR 기능에서 고려할 예외 상황은 다음과 같다.

| 상황              | 처리 방식                      |
| --------------- | -------------------------- |
| 이미지가 업로드되지 않음   | 업로드 안내 메시지 표시              |
| OCR 결과가 비어 있음   | 직접 입력하거나 다른 이미지를 업로드하도록 안내 |
| 이미지 형식이 올바르지 않음 | 지원 형식 안내                   |
| 글자가 흐리거나 작음     | OCR 결과가 부정확할 수 있음을 안내      |

---

## 11. 예외 처리 설계

| 예외 상황         | 처리 방식                                              |
| ------------- | -------------------------------------------------- |
| 이미지가 업로드되지 않음 | “이미지를 업로드해주세요.” 메시지 표시                             |
| OCR 결과가 비어 있음 | “텍스트를 추출하지 못했습니다. 직접 입력하거나 다른 이미지를 사용해주세요.” 메시지 표시 |
| 제목이 비어 있음     | 내용 앞부분을 제목으로 자동 설정                                 |
| 저장할 내용이 비어 있음 | 저장하지 않고 경고 메시지 표시                                  |
| 저장된 데이터가 없음   | “아직 저장된 항목이 없습니다.” 메시지 표시                          |
| 검색어가 비어 있음    | “검색어를 입력해주세요.” 메시지 표시                              |
| 검색 결과가 없음     | “관련 결과가 없습니다.” 메시지 표시                              |

## 12. 품질 관리 관점 설계

| 품질 요소   | 설계 반영                                |

| 확장성     | 향후 모바일 공유 기능, 이미지 원본 저장, 태그 기능 추가 가능 |
| 테스트 용이성 | 기능별 함수 분리를 통해 개별 테스트 가능              |

## 13. 향후 확장 가능성

이번 MVP에서는 구현 범위를 줄였지만, 실제 서비스로 발전시킬 경우 다음 기능을 추가할 수 있다.

1. 모바일 앱 공유 기능
2. 갤러리 자동 연동
3. 이미지 원본 저장 및 미리보기
4. 사용자 계정 및 로그인
5. 클라우드 동기화
6. 태그 자동 추천
7. 카테고리 분류
8. 검색 결과 하이라이트
9. 저장 항목 삭제 및 수정
10. 이미지와 텍스트를 함께 보는 상세 페이지

## 14. 설계 단계의 의의

이번 설계 단계에서는 요구사항을 바로 코드로 구현하지 않고, 화면 흐름, 데이터 구조, 모듈 구조, 검색 방식, 예외 처리 방식을 먼저 정의하였다.

이를 통해 개발 과정에서 어떤 기능을 먼저 구현해야 하는지 명확해졌고, 기능별 책임을 나누어 유지보수하기 쉬운 구조를 만들 수 있었다.

또한 초기 아이디어였던 모바일 앱 공유 기능을 바로 구현하지 않고, 웹 기반 MVP로 범위를 줄인 것은 프로세스 적용의 중요한 결과이다. 이는 바이브코딩 과정에서도 AI가 제안하거나 사용자가 떠올린 기능을 모두 구현하는 것이 아니라, 요구사항 분석과 설계를 통해 핵심 기능을 선별해야 한다는 점을 보여준다.
