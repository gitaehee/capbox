# 2026-05-20 토의 기록: 시스템 설계

## 1. 오늘의 목표

5월 19일에 확정한 요구사항을 바탕으로,
SnapText Archive의 화면 흐름, 데이터 구조, 모듈 구조, 처리 흐름을 설계한다.

이번 단계의 목적은 바로 구현에 들어가기 전에
기능별 책임을 나누고, 전체 시스템의 동작 방식을 명확히 정의하는 것이다.

## 2. 설계 방향

이번 프로젝트는 과제 기간 안에 완성 가능한 MVP를 목표로 하므로,
복잡한 서버-클라이언트 구조 대신 Streamlit 기반 단일 웹앱 구조로 설계한다.

초기 아이디어에는 모바일 앱 공유 기능과 클라우드 저장 기능이 포함되어 있었지만,
현재 MVP에서는 웹 이미지 업로드, OCR, 로컬 저장, 유사도 검색 기능에 집중한다.

## 3. 화면 흐름 논의

사용자 화면은 다음 흐름으로 구성하기로 하였다.

1. 사용자가 스크린샷 이미지를 업로드한다.
2. 시스템이 OCR을 수행하여 이미지 속 텍스트를 추출한다.
3. 사용자는 추출된 텍스트를 확인하고 필요하면 수정한다.
4. 사용자가 저장 버튼을 누르면 텍스트가 SQLite DB에 저장된다.
5. 사용자는 저장된 아카이브 목록을 확인할 수 있다.
6. 사용자가 검색어를 입력하면 저장된 텍스트 중 유사한 결과를 확인할 수 있다.

## 4. 데이터 구조 논의

스크린샷 원본 이미지를 저장하면 저장 용량과 개인정보 관리 부담이 커질 수 있으므로,
이번 MVP에서는 이미지 원본보다는 추출된 텍스트 중심으로 저장한다.

저장 항목은 다음과 같이 정하였다.

- id
- title
- content
- created_at

title은 사용자가 직접 입력하거나,
입력하지 않을 경우 content의 앞부분을 자동 제목처럼 사용할 수 있도록 한다.

## 5. 모듈 구조 논의

기능별 책임을 나누기 위해 다음과 같은 모듈 구조를 사용하기로 하였다.

- app.py: Streamlit 화면 및 전체 흐름 제어
- ocr_utils.py: 이미지에서 텍스트를 추출하는 OCR 기능
- db.py: SQLite 데이터베이스 초기화, 저장, 조회 기능
- search_utils.py: 저장된 텍스트와 검색어 간 유사도 계산 기능

이렇게 모듈을 분리하면 특정 기능을 수정할 때 전체 코드를 고치지 않아도 되며,
테스트와 품질 관리도 더 쉽게 진행할 수 있다.

## 6. 검색 방식 논의

검색 기능은 두 단계로 구현하기로 하였다.

1차 구현에서는 단순 키워드 검색을 적용한다.
이 방식은 구현이 쉽고 동작 확인이 빠르지만,
사용자가 정확한 단어를 입력하지 않으면 원하는 결과를 찾기 어렵다는 한계가 있다.

2차 구현에서는 sentence-transformers 기반 임베딩과 cosine similarity를 활용하여
검색어와 저장된 텍스트의 의미적 유사도를 계산한다.

이를 통해 사용자가 정확한 문장을 기억하지 못하고
뭉뚱그린 표현으로 검색해도 관련 내용을 찾을 수 있도록 한다.

## 7. 오늘의 결정 사항

오늘 설계 단계에서 다음 사항을 결정하였다.

1. Streamlit 기반 단일 웹앱 구조로 구현한다.
2. SQLite를 사용하여 추출 텍스트를 로컬에 저장한다.
3. 기능별 코드를 app.py, ocr_utils.py, db.py, search_utils.py로 분리한다.
4. MVP에서는 이미지 원본 저장보다 텍스트 저장에 집중한다.
5. 검색 기능은 키워드 검색을 먼저 구현하고, 이후 유사도 검색으로 확장한다.
