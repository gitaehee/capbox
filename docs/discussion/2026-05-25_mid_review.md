# 2026-05-25 토의 기록: 중간 점검 및 구현 리뷰

## 1. 오늘의 목표

5월 21일부터 5월 24일까지 구현한 기능을 중간 점검하고, 요구사항 대비 구현 상태를 확인한다.

오늘은 새로운 기능을 추가하기보다, 지금까지 구현한 기능이 요구사항과 설계에 맞게 진행되고 있는지 검토하고, 발견된 한계와 다음 개선 방향을 정리한다.

## 2. 현재까지의 구현 내용

현재까지 구현된 기능은 다음과 같다.

1. Streamlit 기본 화면 구성
2. 스크린샷 이미지 업로드
3. 업로드 이미지 미리보기
4. OCR 텍스트 추출
5. OCR 결과 확인 및 수정
6. SQLite 저장
7. 저장된 아카이브 목록 조회
8. 키워드 검색

이를 통해 사용자는 이미지를 업로드하고, OCR로 텍스트를 추출한 뒤, 수정한 내용을 저장하고, 저장된 항목을 키워드로 검색할 수 있게 되었다.

## 3. 요구사항 대비 점검

요구사항 문서에서 정의한 기능 요구사항과 현재 구현 상태를 비교하였다.

| 요구사항 ID | 요구사항 | 현재 상태 |
|---|---|---|
| FR-01 | 이미지 업로드 | 구현 완료 |
| FR-02 | 이미지에서 텍스트 추출 | 구현 완료 |
| FR-03 | OCR 결과 확인 및 수정 | 구현 완료 |
| FR-04 | 텍스트 저장 | 구현 완료 |
| FR-05 | 저장 목록 조회 | 구현 완료 |
| FR-06 | 검색어 입력 | 구현 완료 |
| FR-07 | 유사한 저장 내용 반환 | 키워드 검색으로 일부 구현 |
| FR-08 | 유사도 점수 표시 | 미구현 |

현재까지 MVP의 핵심 흐름은 동작하지만, “뭉뚱그린 검색어로 유사한 내용을 찾는 기능”은 아직 완전히 구현되지 않았다.

## 4. 테스트 중 발견한 사항

OCR 기능 테스트 중 일부 한글 단어가 잘못 인식되는 문제가 있었다.

예를 들어 원본 텍스트와 OCR 결과는 다음과 같았다.

| 원본 | OCR 결과 |
|---|---|
| 젤 | 절 |
| 살몬 | 살문 |
| 댓글 | 대글 |

이 문제는 OCR의 일반적인 한계로 볼 수 있다.  
따라서 OCR 결과를 사용자가 직접 수정한 뒤 저장할 수 있도록 한 설계가 실제로 필요하다는 점을 확인하였다.

## 5. 현재 구조의 장점

현재 구조의 장점은 다음과 같다.

1. 기능별 파일이 분리되어 있다.
2. OCR 로직이 `ocr_utils.py`로 분리되어 있다.
3. DB 로직이 `db.py`로 분리되어 있다.
4. 검색 로직이 `search_utils.py`로 분리되어 있다.
5. 화면 흐름은 `app.py`에서 확인할 수 있다.

이 구조 덕분에 이후 유사도 검색 기능을 추가할 때도 기존 OCR 또는 DB 코드를 크게 수정하지 않고 `search_utils.py`를 중심으로 확장할 수 있다.

## 6. 현재 한계

현재 구현의 한계는 다음과 같다.

1. 키워드 검색은 정확한 단어가 포함되어야만 결과를 찾을 수 있다.
2. OCR이 잘못 인식한 단어는 키워드 검색에서 검색되지 않을 수 있다.
3. 저장된 항목을 삭제하거나 수정하는 기능이 없다.
4. 유사도 점수가 표시되지 않는다.
5. 이미지 원본은 저장하지 않기 때문에 원본 스크린샷을 다시 확인할 수 없다.

## 7. 다음 작업

다음 단계에서는 키워드 검색의 한계를 보완하기 위해 유사도 검색 기능을 구현한다.

검색어와 저장된 텍스트를 임베딩 벡터로 변환하고, cosine similarity를 계산하여 의미적으로 관련성이 높은 결과를 반환하는 방식으로 개선할 예정이다.

## 8. 중간 점검에서 얻은 교훈

이번 중간 점검을 통해 기능을 한 번에 완성하려 하기보다, 요구사항을 기준으로 구현 상태를 주기적으로 점검하는 것이 중요하다는 점을 확인하였다.

특히 바이브코딩으로 코드를 빠르게 작성할 수는 있었지만, 생성된 기능이 실제 요구사항을 충분히 만족하는지 확인하는 과정은 별도로 필요했다.

따라서 바이브코딩을 사용할 때도 요구사항 문서, 설계 문서, 구현 점검, 테스트 기록을 함께 관리해야 한다는 교훈을 얻었다.
