# 2026-05-23 토의 기록: SQLite 저장 기능 구현

## 1. 오늘의 목표

5월 22일에 구현한 OCR 기능을 바탕으로,
사용자가 확인 및 수정한 OCR 결과를 SQLite 데이터베이스에 저장하는 기능을 구현한다.

오늘의 목표는 OCR 결과를 단순히 화면에 표시하는 것에서 나아가,
텍스트 아카이빙 서비스의 핵심 기능인 “저장” 흐름을 완성하는 것이다.

## 2. 구현 범위

오늘 구현한 범위는 다음과 같다.

1. SQLite 데이터베이스 초기화
2. archive 테이블 생성
3. OCR 결과 텍스트 저장 기능 구현
4. 저장 제목 입력 기능 구현
5. 제목이 비어 있을 경우 자동 제목 생성
6. 저장된 아카이브 목록 조회
7. 빈 텍스트 저장 방지

검색 기능은 아직 구현하지 않고,
저장 및 조회 기능이 정상적으로 동작하는지 확인하는 데 집중하였다.

## 3. 설계 반영 내용

5월 20일 설계 문서에서는 DB 관련 기능을 `db.py`로 분리하기로 하였다.

오늘 구현에서는 이 설계를 반영하여,
SQLite 초기화, 저장, 조회 기능을 모두 `db.py`에 작성하였다.
`app.py`는 화면 구성과 사용자 흐름을 담당하고,
DB 접근 로직은 `db.py`가 담당하도록 역할을 분리하였다.

이를 통해 화면 코드와 데이터 저장 코드가 분리되어,
향후 검색 기능이나 삭제 기능을 추가할 때 유지보수가 쉬운 구조가 되었다.

## 4. 데이터 구조

오늘 구현한 `archive` 테이블은 다음 컬럼으로 구성하였다.

| 컬럼명 | 설명 |
|---|---|
| id | 저장 항목의 고유 번호 |
| title | 저장 항목 제목 |
| content | OCR로 추출 및 수정된 텍스트 |
| created_at | 저장 시각 |

이미지 원본은 저장하지 않고,
OCR 결과 텍스트와 사용자가 수정한 텍스트만 저장하였다.

이는 저장 용량을 줄이고,
스크린샷에 포함될 수 있는 개인정보 보관 부담을 낮추기 위한 결정이다.

## 5. 확인 결과

테스트용 스크린샷 이미지를 업로드하고 OCR을 실행한 뒤,
추출된 텍스트를 수정하여 제목과 함께 저장할 수 있었다.

저장 후에는 화면 하단의 저장된 아카이브 목록에
제목, 저장 시각, 내용이 표시되는 것을 확인하였다.

또한 제목을 입력하지 않은 경우에는 내용 앞부분을 제목으로 자동 사용하도록 처리하였다.

## 6. 발견한 한계

현재는 저장된 항목을 삭제하거나 수정하는 기능이 없다.
따라서 잘못 저장한 항목이 있을 경우 DB 파일을 직접 삭제하거나 이후 기능을 추가해야 한다.

또한 현재 검색 기능이 구현되지 않았기 때문에,
저장된 데이터는 목록에서 직접 확인하는 방식으로만 조회할 수 있다.
